# 05 — Disambiguation Rules

Apply in this order when goals compete.

- **Intervening vs. Problem-solving:** If the paper describes changing the model
  and then reflecting that change in a schema, system, artifact, or
  implementation, code For Intervening. If it only uses the model to identify
  shortcomings, inconsistencies, problems, or possible improvements, code For
  Problem-solving. If the same evidence only shows implementation, code For
  Intervening, not For Problem-solving. Code both only when separate evidence
  shows the model also supported decision-making, analysis, evaluation, or option
  comparison in addition to implementation.

- **Communicating vs. Documenting:** Synchronous/live use (meetings, real-time
  explanation) -> Communicating. Asynchronous/persistent use (reference consumed
  later by absent parties) -> Documenting. If both clearly occur, code both.

- **Understanding vs. Communicating:** Understanding is building knowledge/meaning
  (may be one person or the modeling team itself). Communicating requires
  conveying meaning between distinct parties. Internal sense-making ->
  Understanding; bridging two groups -> Communicating.

- **Understanding vs. Learning:** Understanding concerns the DOMAIN. Learning
  concerns the MODELING activity (method/tool/notation). Work framed as a test
  case to learn whether/how modeling works -> Learning.

- **Goals are NOT mutually exclusive:** assign every goal with sufficient
  evidence; double-coding is correct when the text genuinely supports each goal.
  Do not double-code one vague sentence into multiple goals.
