# Disambiguation Rules (all pairs, in order)

Apply in this order when goals compete. Each rule also appears in the individual
goal files; this is the combined reference.

1. **Intervening vs. Problem-solving:** Change reflected in a built
   schema/system/artifact/implementation -> Intervening. Only identifying
   shortcomings/problems/improvements -> Problem-solving. Same evidence showing
   only implementation -> Intervening, not both. Code both only with separate
   evidence of decision-making/analysis in addition to implementation.

2. **Communicating vs. Documenting:** Synchronous/live (meetings, real-time
   explanation) -> Communicating. Asynchronous/persistent (reference consumed
   later by absent parties) -> Documenting. Both clearly occur -> code both.

3. **Understanding vs. Communicating:** Building knowledge/meaning (one person or
   the modeling team) -> Understanding. Conveying meaning between distinct parties
   -> Communicating.

4. **Understanding vs. Learning:** Domain knowledge -> Understanding. Modeling
   activity (method/tool/notation) -> Learning. Test case to learn whether/how
   modeling works -> Learning.

5. **Goals are NOT mutually exclusive:** assign every goal with sufficient
   evidence; double-coding is correct when the text genuinely supports each.
   Do not double-code one vague sentence into multiple goals.
